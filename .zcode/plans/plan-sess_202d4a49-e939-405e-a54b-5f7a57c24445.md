## 优化 Worker java.ajax() 死锁

### 根因

C++ 桥 spin loop 中用 `uv_default_loop()` 获取 libuv 事件循环，但这拿到的是 C++ 侧的默认 loop，**不是 ArkTS Worker 运行时实际使用的 loop**。RCP 的 Promise resolve 回调挂在 ArkTS 的 loop 上，所以 `uv_run(uv_default_loop(), UV_RUN_NOWAIT)` 永远 pump 不到 RCP 回调，导致 `req->completed` 无法被置位，spin loop 跑满 30s 超时。

### 修复方案

用 `napi_get_uv_event_loop(g_main_env, &loop)` 替代 `uv_default_loop()`，获取 ArkTS Worker 运行时实际绑定的 libuv loop。HarmonyOS NAPI 从 API 10 起支持此接口。

### 改动文件

**`libraries/quickjs/src/napi_bridge.cpp`** — 两处 spin loop（`js_http_get` 和 `js_http_post`）：

```cpp
// 之前（错误）：
uv_loop_t *loop = uv_default_loop();
if (loop) { uv_run(loop, UV_RUN_NOWAIT); }

// 之后（正确）：
uv_loop_t *loop = nullptr;
if (g_main_env && napi_get_uv_event_loop(g_main_env, &loop) == napi_ok && loop) {
  uv_run(loop, UV_RUN_NOWAIT);
}
```

同时把 `sleep 1ms` 改为 `sleep 2ms`，减少不必要的 CPU 空转（`UV_RUN_NOWAIT` 已经 non-blocking，不需要每毫秒轮询）。

### 预期效果

- `uv_run` pump 的是 ArkTS Worker 运行时的真实 libuv loop
- RCP `fetch()` 的 Promise resolve 回调能被正确处理
- `handleHttpRequest` 的 `await` 完成后调用 `onHttpResponse` -> `req->completed = true` -> spin loop 立即退出
- 预期 java.ajax() 从 7-33 秒降到 <1 秒
